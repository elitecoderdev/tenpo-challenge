Modelos & Entidades

Entity JPA, tabla en PostgreSQL, índices

Entidad Transaction

@Entity
@Table(name = "transactions")
public class Transaction {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Integer id;

    @Column(name = "amount_in_pesos", nullable = false)
    private int amountInPesos;

    @Column(nullable = false, length = 160)
    private String merchant;

    @Column(name = "customer_name",
            nullable = false, length = 120)
    private String customerName;          // ← valor original

    @Column(name = "customer_name_normalized",
            nullable = false, length = 120)
    private String customerNameNormalized; // ← lowercase + trim

    @Column(name = "transaction_date", nullable = false)
    private LocalDateTime transactionDate;

    @PrePersist @PreUpdate
    void normalizeCustomerName() {
        customerNameNormalized = canonicalize(customerName);
    }

    public static String canonicalize(String v) {
        return v.trim()
                .replaceAll("\\s+", " ")
                .toLowerCase(Locale.ROOT);
    }
}

Tabla en PostgreSQL

CREATE TABLE transactions (
  id INTEGER GENERATED BY DEFAULT
             AS IDENTITY PRIMARY KEY,
  amount_in_pesos  INTEGER      NOT NULL
                   CHECK (amount_in_pesos >= 0),
  merchant         VARCHAR(160) NOT NULL,
  customer_name    VARCHAR(120) NOT NULL,
  customer_name_normalized VARCHAR(120) NOT NULL,
  transaction_date TIMESTAMP   NOT NULL
);

-- Índice para filtrar/contar por cliente
CREATE INDEX idx_transactions_customer_name_normalized
  ON transactions (customer_name_normalized);

-- Índice para ordenamiento por fecha
CREATE INDEX idx_transactions_transaction_date
  ON transactions (transaction_date DESC, id DESC);
✅

Decisión de diseño: Se guarda el nombre del cliente en dos columnas: customer_name (valor original para mostrar) y customer_name_normalized (para búsquedas y conteo de cuota). Esto garantiza que "Juan Torres", " juan torres " y "JUAN TORRES" cuenten como el mismo cliente.

Servicios & Lógica de Negocio

TransactionService — el núcleo de la aplicación

Reglas de Negocio

Cuota de 100 transacciones: Ningún cliente puede tener más de 100 transacciones activas. La validación usa el nombre normalizado. Al crear: cuenta actual >= 100 → 409 Conflict. Al editar: si el nombre cambia, cuenta del nuevo nombre >= 100 → 409 Conflict.
Monto no negativo: amountInPesos debe ser ≥ 0. Validado en DTO (Bean Validation) y reforzado con CHECK en la base de datos.
Fecha no futura: transactionDate debe ser pasada o presente. Validada en DTO con @PastOrPresent.
Normalización de texto: Merchant y customerName son saneados: trim() + collapsado de espacios internos (\\s+ → " ") antes de persistir.
Nombres de cliente canonicalizados: customerNameNormalized se recalcula automáticamente en @PrePersist y @PreUpdate mediante JPA lifecycle hooks.
Zonas horarias: Hibernate configurado con UTC (hibernate.jdbc.time_zone: UTC). Jackson no ajusta fechas al contexto. Las fechas se almacenan y devuelven en UTC.

Implementación de métodos principales

@Service
public class TransactionService {
    private static final int MAX_TRANSACTIONS_PER_CUSTOMER = 100;

    // Lista todas o filtra por customerName (normalizado)
    @Transactional(readOnly = true)
    public List<TransactionResponse> listTransactions(String customerName) { ... }

    // Crea: normaliza, valida cuota, persiste
    @Transactional
    public TransactionResponse createTransaction(TransactionRequest req) {
        String sanitizedName = sanitizeText(req.customerName());
        ensureTransactionQuota(sanitizedName, null);  // verifica cuota
        Transaction t = new Transaction(...);
        return mapper.toResponse(repository.save(t));
    }

    // Actualiza: si cambia el cliente, re-valida cuota
    @Transactional
    public TransactionResponse updateTransaction(Integer id, TransactionRequest req) {
        Transaction existing = findOrThrow(id);
        ensureTransactionQuota(sanitizedName, existing); // ignora si cliente no cambió
        // actualiza campos...
        return mapper.toResponse(repository.save(existing));
    }

    private void ensureTransactionQuota(String name, Transaction existing) {
        String normalized = Transaction.canonicalize(name);
        boolean clientChanged = existing == null
            || !normalized.equals(existing.getCustomerNameNormalized());
        if (clientChanged) {
            long count = repository.countByCustomerNameNormalized(normalized);
            if (count >= MAX_TRANSACTIONS_PER_CUSTOMER)
                throw new BusinessRuleException(HttpStatus.CONFLICT,
                    "Customer has reached the maximum of 100 transactions.");
        }
    }

    private String sanitizeText(String v) {
        return v.trim().replaceAll("\\s+", " ");
    }
}

Validaciones & DTOs

Records Java + Bean Validation + Zod en frontend

TransactionRequest (record)

public record TransactionRequest(
    @PositiveOrZero(message = "Transaction amount cannot be negative.")
    @Max(value = Integer.MAX_VALUE, message = "Amount is too large.")
    long amountInPesos,  // long para recibir el rango completo

    @NotBlank(message = "Merchant is required.")
    @Size(max = 160, message = "Merchant cannot exceed 160 chars.")
    String merchant,

    @NotBlank(message = "Customer name is required.")
    @Size(max = 120, message = "Customer name cannot exceed 120 chars.")
    String customerName,

    @NotNull(message = "Transaction date is required.")
    @PastOrPresent(message = "Transaction date cannot be in the future.")
    LocalDateTime transactionDate
) {}
💡

Detalle: amountInPesos se recibe como long en el DTO para poder validar con @Max(Integer.MAX_VALUE) sin overflow. El service convierte a int antes de persistir. Esto evita excepciones no controladas por desbordamiento.

Implementación — Fixed Window Counter

// FixedWindowCounter (clase interna en ClientRateLimiter)
private static final class FixedWindowCounter {
    private Instant windowStart;
    private int consumedTokens;

    synchronized RateLimitDecision consume(Instant now, int capacity, Duration duration) {
        // ¿Estamos fuera de la ventana? → reset
        if (windowStart == null || !now.isBefore(windowStart.plus(duration))) {
            windowStart = now;
            consumedTokens = 0;
        }
        // ¿Hay tokens disponibles?
        if (consumedTokens < capacity) {
            consumedTokens++;
            return RateLimitDecision.allowed(capacity - consumedTokens, Duration.ZERO);
        }
        // Bloqueado: calcular retry-after
        Duration retryAfter = Duration.between(now, windowStart.plus(duration));
        long retryAfterSeconds = Math.max(1, (retryAfter.toMillis() + 999) / 1_000);
        return RateLimitDecision.blocked(Duration.ofSeconds(retryAfterSeconds));
    }
}

// Caffeine guarda un FixedWindowCounter por IP de cliente
Cache<String, FixedWindowCounter> cache = Caffeine.newBuilder()
    .expireAfterAccess(duration.multipliedBy(2))
    .build();

El catch-all devuelve mensajes genéricos en 500. No se exponen stack traces, nombres de clases ni paths internos.

CHECK en DB

CHECK (amount_in_pesos >= 0) en la tabla. Defensa en profundidad a nivel base de datos.

DDL validate

Hibernate con ddl-auto: validate. Nunca modifica el schema en arranque. Flyway es el único que migra.

Max header size

Limitado a 64KB via server.max-http-request-header-size.

Actuator limitado

Solo expone /actuator/health y /actuator/info. No expone env, beans ni otras rutas sensibles.

Estado & Queries

TanStack Query — server state management
// App.tsx — Query principal
const { data: transactions = [], error, isFetching, isLoading } =
  useQuery(transactionsQueryOptions)

// Mutation de creación con actualización optimista del cache
const createMutation = useMutation({
  mutationFn: createTransaction,
  onSuccess: (createdTransaction) => {
    // Actualiza el cache INMEDIATAMENTE sin refetch
    queryClient.setQueryData<Transaction[]>(
      transactionKeys.all,
      (current = []) => sortTransactions([createdTransaction, ...current])
    )
  },
})

// Mutation de actualización
const updateMutation = useMutation({
  mutationFn: ({ id, data }) => updateTransaction(id, data),
  onSuccess: (updated) => {
    queryClient.setQueryData<Transaction[]>(
      transactionKeys.all,
      (current = []) =>
        sortTransactions(current.map(t => t.id === updated.id ? updated : t))
    )
  },
})

// Mutation de eliminación
const deleteMutation = useMutation({
  mutationFn: deleteTransaction,
  onSuccess: (_, deletedId) => {
    queryClient.setQueryData<Transaction[]>(
      transactionKeys.all,
      (current = []) => current.filter(t => t.id !== deletedId)
    )
  },
})
⚡

Cache Optimista: Las mutations actualizan el cache local de React Query directamente en onSuccess, sin disparar un refetch. Esto da una UX instantánea. Si hay error, el cache queda stale hasta el próximo refetch.

SOLID, DRY, Seguridad y Rendimiento

Análisis de principios de diseño aplicados en este proyecto

Principios SOLID — Backend

S — Single Responsibility: TransactionController solo maneja HTTP (routing, status codes, headers). TransactionService solo maneja reglas de negocio. TransactionMapper solo convierte entidades ↔ DTOs. RateLimitFilter solo aplica el límite en el pipeline. ClientRateLimiter solo implementa el algoritmo. Cada clase tiene exactamente una razón para cambiar.
O — Open/Closed: GlobalExceptionHandler está abierto a nuevos tipos de excepción (nuevo método @ExceptionHandler) sin modificar los handlers existentes. BusinessRuleException permite que cualquier regla de negocio lance con cualquier HTTP status sin cambiar el handler.
L — Liskov Substitution: ResourceNotFoundException y BusinessRuleException extienden RuntimeException. Son intercambiables en cualquier catch (RuntimeException) sin romper el comportamiento.
I — Interface Segregation: TransactionService solo usa los métodos que necesita de TransactionRepository: save(), findById(), deleteById(), findAll(Sort), y las dos queries derivadas. No depende del contrato completo de JpaRepository.
D — Dependency Inversion: Todas las dependencias se inyectan por constructor (sin @Autowired en campos). TransactionController depende del bean TransactionService, no de la clase concreta. RateLimitFilter recibe ClientRateLimiter y ClientKeyResolver como beans inyectados.

Principios SOLID — Frontend

S — Single Responsibility: TransactionForm solo gestiona estado del formulario y rendering (sin llamadas a API). TransactionList solo renderiza tarjetas (sin fetch, sin mutaciones). formatters.ts solo formatea moneda. Modal.tsx solo gestiona apertura/cierre accesible.
O — Open/Closed: Nuevos campos en el formulario se agregan extendiendo el schema Zod y añadiendo un <input> — sin cambiar la estructura del formulario. Nuevas mutaciones se agregan como hooks independientes en App.tsx.
D — Dependency Inversion: Los componentes reciben datos y callbacks como props — no llaman a APIs directamente. TransactionForm recibe onSubmit, no importa createTransaction.

Correcciones DRY aplicadas

lib/formatters.ts — Formateador CLP compartido

Problema: Intl.NumberFormat('es-CL', { style: 'currency', currency: 'CLP', maximumFractionDigits: 0 }) se instanciaba independientemente en App.tsx y TransactionList.tsx.

Solución: Extraído a src/lib/formatters.ts como formatCLP(amount: number): string. Constante a nivel de módulo — creada una vez al importar, no en cada render.

transactionKeys factory (queries.ts)

Todas las operaciones de cache de React Query referencian transactionKeys.all. Cambiar la clave del cache requiere editar una sola línea.

applyRequest() — Backend

El método privado applyRequest() en TransactionService es compartido por create() y update(). Evita duplicar la lógica de sanitización, conversión y mapeo de campos.

Rendimiento — decisiones clave

@Transactional(readOnly=true)

Aplicado en findAll() y findById(). Hibernate omite el dirty-checking. Permite routing a réplica de lectura si se configura.

Índices compuestos en DB

(transaction_date DESC, id DESC) cubre el ORDER BY. (customer_name_normalized) hace el conteo de cuota O(log n). Sin full-table scans.

customerNameNormalized pre-computado

Calculado en escritura (@PrePersist). En lectura el índice se usa directamente sin llamada a LOWER(), que invalidaría el índice.

Caffeine in-memory

Lecturas del rate limiter en sub-microsegundos. Sin round-trip a Redis para el scope de un challenge de un nodo.

Fetch una vez, filtrar local

El frontend filtra por cliente localmente (zero requests). useDeferredValue mantiene el input responsivo mientras la lista re-renderiza.

Cache optimista post-mutación

queryClient.setQueryData() en lugar de refetch(). Feedback visual inmediato. Preserva las 3 solicitudes/min del rate limit.

Rendimiento Frontend — optimizaciones adicionales

useMemo para valores derivados

visibleTransactions, totalAmount y uniqueCustomers se calculan con useMemo en App.tsx. Solo se recomputan cuando cambian sus dependencias, evitando recálculos en re-renders no relacionados.

useCallback para event handlers

Todos los callbacks de eventos en App.tsx están envueltos en useCallback. Las referencias de función son estables entre renders, lo que evita re-renders innecesarios en los componentes hijos que los reciben como props.

React.memo en TransactionList y TransactionForm

Ambos componentes están envueltos en React.memo. Se saltean el re-render cuando sus props no cambian, lo cual es especialmente beneficioso cuando el estado del componente padre cambia por razones no relacionadas (ej. apertura del modal).

Code splitting con manualChunks

vite.config.ts define manualChunks para dividir las librerías de vendor en chunks separados: vendor-react, vendor-query, vendor-utils, vendor-forms. El código de la app cambia frecuentemente; los vendors rara vez. El navegador puede reutilizar los chunks de vendor cacheados entre deploys.

Puntos Clave para una Conversación Senior

Decisiones de diseño notables y trade-offs
Doble columna para nombre de cliente: Guardar customerName (original) y customerNameNormalized (canónico) resuelve elegantemente el conteo de cuota insensible a mayúsculas/espacios, sin perder el valor original para mostrar al usuario.
long en DTO, int en entidad: Recibir amountInPesos como long en el record permite validar con @Max(Integer.MAX_VALUE) sin overflow silencioso. El service hace el cast a int solo tras validar. Previene una clase entera de bugs sutiles.
Rate limiter in-memory (Caffeine) es intencional: No es distribuido. Funciona para un solo nodo. En producción multi-instancia se necesitaría Redis. El diseño lo hace reemplazable (ClientRateLimiter + RateLimitFilter son independientes).
Fixed Window vs Token Bucket: Se eligió Fixed Window por simplicidad e implementación directa. Trade-off: puede permitir ráfagas al final/inicio de ventanas. Token Bucket sería más suave pero más complejo.
Flyway + ddl-auto: validate: Pattern recomendado para producción. Flyway es la fuente de verdad del schema. Hibernate valida pero nunca modifica. Evita sorpresas con drops/creates automáticos.
Cache optimista en frontend: Las mutations actualizan el cache de React Query directamente. Esto evita un round-trip al servidor tras cada operación, dando feedback visual inmediato.
Package by feature: Todo el dominio de transacciones en un paquete. Facilita cohesión, navegación y el eventual split en microservicio si el proyecto crece.
Records Java para DTOs: Inmutables por defecto, sin boilerplate (no getters/setters/equals/hashCode). Expresan claramente la intención de "data transfer only".
