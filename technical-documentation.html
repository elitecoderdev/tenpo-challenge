Models & Entities

JPA Entity, PostgreSQL table, indexes

Transaction Entity

@Entity
@Table(name = "transactions")
public class Transaction {

    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Integer id;

    @Column(name = "amount_in_pesos", nullable = false)
    private int amountInPesos;

    @Column(nullable = false, length = 160)
    private String merchant;

    @Column(name = "customer_name",
            nullable = false, length = 120)
    private String customerName;       // ← original value

    @Column(name = "customer_name_normalized",
            nullable = false, length = 120)
    private String customerNameNormalized; // ← lowercase + trim

    @Column(name = "transaction_date", nullable = false)
    private LocalDateTime transactionDate;

    @PrePersist @PreUpdate
    void normalizeCustomerName() {
        customerNameNormalized = canonicalize(customerName);
    }

    public static String canonicalize(String v) {
        return v.trim()
                .replaceAll("\\s+", " ")
                .toLowerCase(Locale.ROOT);
    }
}

PostgreSQL Table

CREATE TABLE transactions (
  id INTEGER GENERATED BY DEFAULT
             AS IDENTITY PRIMARY KEY,
  amount_in_pesos  INTEGER      NOT NULL
                   CHECK (amount_in_pesos >= 0),
  merchant         VARCHAR(160) NOT NULL,
  customer_name    VARCHAR(120) NOT NULL,
  customer_name_normalized VARCHAR(120) NOT NULL,
  transaction_date TIMESTAMP   NOT NULL
);

-- Index for filtering/counting by customer
CREATE INDEX idx_transactions_customer_name_normalized
  ON transactions (customer_name_normalized);

-- Index for date ordering
CREATE INDEX idx_transactions_transaction_date
  ON transactions (transaction_date DESC, id DESC);
✅

Design decision: The customer name is stored in two columns: customer_name (original for display) and customer_name_normalized (for searches and quota counting). This ensures "Juan Torres", " juan torres " and "JUAN TORRES" are counted as the same customer.

Services & Business Logic

TransactionService — the application core

Business Rules

100-transaction quota: No customer can have more than 100 active transactions. Validation uses the normalised name. On create: current count >= 100 → 409 Conflict. On edit: if name changes, count of new name >= 100 → 409 Conflict.
Non-negative amount: amountInPesos must be ≥ 0. Validated in the DTO (Bean Validation) and enforced with a CHECK constraint in the database.
No future date: transactionDate must be past or present. Validated in the DTO with @PastOrPresent.
Text normalisation: Merchant and customerName are sanitised: trim() + internal whitespace collapse (\\s+ → " ") before persisting.
Canonicalised customer names: customerNameNormalized is automatically recalculated in @PrePersist and @PreUpdate via JPA lifecycle hooks.
Time zones: Hibernate configured with UTC (hibernate.jdbc.time_zone: UTC). Jackson does not adjust dates to context. Dates are stored and returned in UTC.

Main method implementation

@Service
public class TransactionService {
    private static final int MAX_TRANSACTIONS_PER_CUSTOMER = 100;

    // Lists all or filters by customerName (normalised)
    @Transactional(readOnly = true)
    public List<TransactionResponse> listTransactions(String customerName) { ... }

    // Create: normalise, check quota, persist
    @Transactional
    public TransactionResponse createTransaction(TransactionRequest req) {
        String sanitizedName = sanitizeText(req.customerName());
        ensureTransactionQuota(sanitizedName, null);  // check quota
        Transaction t = new Transaction(...);
        return mapper.toResponse(repository.save(t));
    }

    // Update: if customer changes, re-check quota
    @Transactional
    public TransactionResponse updateTransaction(Integer id, TransactionRequest req) {
        Transaction existing = findOrThrow(id);
        ensureTransactionQuota(sanitizedName, existing); // ignored if customer unchanged
        // update fields...
        return mapper.toResponse(repository.save(existing));
    }

    private void ensureTransactionQuota(String name, Transaction existing) {
        String normalized = Transaction.canonicalize(name);
        boolean clientChanged = existing == null
            || !normalized.equals(existing.getCustomerNameNormalized());
        if (clientChanged) {
            long count = repository.countByCustomerNameNormalized(normalized);
            if (count >= MAX_TRANSACTIONS_PER_CUSTOMER)
                throw new BusinessRuleException(HttpStatus.CONFLICT,
                    "Customer has reached the maximum of 100 transactions.");
        }
    }

    private String sanitizeText(String v) {
        return v.trim().replaceAll("\\s+", " ");
    }
}

Validation & DTOs

Java Records + Bean Validation + Zod on frontend

TransactionRequest (record)

public record TransactionRequest(
    @PositiveOrZero(message = "Transaction amount cannot be negative.")
    @Max(value = Integer.MAX_VALUE, message = "Amount is too large.")
    long amountInPesos,  // long to accept full range for validation

    @NotBlank(message = "Merchant is required.")
    @Size(max = 160, message = "Merchant cannot exceed 160 chars.")
    String merchant,

    @NotBlank(message = "Customer name is required.")
    @Size(max = 120, message = "Customer name cannot exceed 120 chars.")
    String customerName,

    @NotNull(message = "Transaction date is required.")
    @PastOrPresent(message = "Transaction date cannot be in the future.")
    LocalDateTime transactionDate
) {}
💡

Detail: amountInPesos is received as long in the DTO to allow validation with @Max(Integer.MAX_VALUE) without silent overflow. The service converts to int only after validation. Prevents an entire class of subtle bugs.

Implementation — Fixed Window Counter

// FixedWindowCounter (inner class in ClientRateLimiter)
private static final class FixedWindowCounter {
    private Instant windowStart;
    private int consumedTokens;

    synchronized RateLimitDecision consume(Instant now, int capacity, Duration duration) {
        // Outside the window? → reset
        if (windowStart == null || !now.isBefore(windowStart.plus(duration))) {
            windowStart = now;
            consumedTokens = 0;
        }
        // Tokens available?
        if (consumedTokens < capacity) {
            consumedTokens++;
            return RateLimitDecision.allowed(capacity - consumedTokens, Duration.ZERO);
        }
        // Blocked: calculate retry-after
        Duration retryAfter = Duration.between(now, windowStart.plus(duration));
        long retryAfterSeconds = Math.max(1, (retryAfter.toMillis() + 999) / 1_000);
        return RateLimitDecision.blocked(Duration.ofSeconds(retryAfterSeconds));
    }
}

// Caffeine keeps one FixedWindowCounter per client IP
Cache<String, FixedWindowCounter> cache = Caffeine.newBuilder()
    .expireAfterAccess(duration.multipliedBy(2))
    .build();

  useQuery(transactionsQueryOptions)

// Create mutation with optimistic cache update
const createMutation = useMutation({
  mutationFn: createTransaction,
  onSuccess: (createdTransaction) => {
    // Updates cache IMMEDIATELY without refetch
    queryClient.setQueryData<Transaction[]>(
      transactionKeys.all,
      (current = []) => sortTransactions([createdTransaction, ...current])
    )
  },

// Update mutation
const updateMutation = useMutation({
  mutationFn: ({ id, data }) => updateTransaction(id, data),
  onSuccess: (updated) => {
    queryClient.setQueryData<Transaction[]>(
      transactionKeys.all,
      (current = []) =>
        sortTransactions(current.map(t => t.id === updated.id ? updated : t))
    )
  },
})

// Delete mutation
const deleteMutation = useMutation({
  mutationFn: deleteTransaction,
  onSuccess: (_, deletedId) => {
    queryClient.setQueryData<Transaction[]>(
      transactionKeys.all,
      (current = []) => current.filter(t => t.id !== deletedId)
    )
  },
})
⚡

Optimistic Cache: Mutations update the local React Query cache directly in onSuccess, without triggering a refetch. This gives instant UX. If there is an error, the cache remains stale until the next refetch. This is a deliberate trade-off for the 3 req/min rate limit.

SOLID, DRY, Security & Performance

Design principles analysis applied across this project

SOLID Principles — Backend

S — Single Responsibility: TransactionController handles only HTTP (routing, status codes, headers). TransactionService handles only business rules. TransactionMapper handles only entity ↔ DTO conversion. RateLimitFilter handles only pipeline enforcement. ClientRateLimiter handles only the algorithm. Each class has exactly one reason to change.
O — Open/Closed: GlobalExceptionHandler is open for extension (new @ExceptionHandler method) without modifying existing handlers. BusinessRuleException lets any business rule throw with any HTTP status without changing the handler.
L — Liskov Substitution: ResourceNotFoundException and BusinessRuleException both extend RuntimeException. They are interchangeable in any catch (RuntimeException) without breaking behaviour.
I — Interface Segregation: TransactionService uses only the methods it needs from TransactionRepository: save(), findById(), deleteById(), findAll(Sort), and the two derived queries. It does not depend on the full JpaRepository contract.
D — Dependency Inversion: All dependencies injected via constructor (no @Autowired on fields). TransactionController depends on the TransactionService bean, not on the concrete class. RateLimitFilter receives ClientRateLimiter and ClientKeyResolver as injected beans.

SOLID Principles — Frontend

S — Single Responsibility: TransactionForm only manages form state and rendering (no API calls). TransactionList only renders cards (no fetching, no mutations). formatters.ts only formats currency. Modal.tsx only manages accessible open/close behaviour.
O — Open/Closed: New form fields are added by extending the Zod schema and adding an <input> — no structural change to the form logic. New mutations are added as independent hooks in App.tsx.
D — Dependency Inversion: Components receive data and callbacks as props — they do not call APIs directly. TransactionForm receives onSubmit, it does not import createTransaction.

DRY fixes applied

lib/formatters.ts — Shared CLP formatter

Problem: Intl.NumberFormat('es-CL', { style: 'currency', currency: 'CLP', maximumFractionDigits: 0 }) was instantiated independently in both App.tsx and TransactionList.tsx.

Fix: Extracted to src/lib/formatters.ts as formatCLP(amount: number): string. Module-level constant — created once at import, not on each render.

transactionKeys factory (queries.ts)

All React Query cache operations reference transactionKeys.all. Changing the cache key requires editing one line.

applyRequest() — Backend

Private method applyRequest() in TransactionService is shared by create() and update(). Avoids duplicating sanitisation, conversion and field-mapping logic.

Performance — key decisions

@Transactional(readOnly=true)

Applied on findAll() and findById(). Hibernate skips dirty-checking. Enables read-replica routing if configured.

Compound DB indexes

(transaction_date DESC, id DESC) covers the ORDER BY. (customer_name_normalized) makes quota count O(log n). No full-table scans.

Pre-computed customerNameNormalized

Computed at write time (@PrePersist). At read time the index is used directly without a LOWER() call, which would invalidate the index.

Caffeine in-memory

Rate limiter reads in sub-microseconds. No Redis round-trip for the single-node challenge scope.

Fetch once, filter locally

Frontend filters by customer locally (zero requests). useDeferredValue keeps the input responsive while the list re-renders.

Optimistic cache post-mutation

queryClient.setQueryData() instead of refetch(). Instant visual feedback. Preserves the 3 req/min budget.

Frontend Performance — additional optimisations

useMemo for derived values

visibleTransactions, totalAmount and uniqueCustomers are computed with useMemo in App.tsx. They recompute only when their dependencies change, avoiding recalculation on unrelated re-renders.

useCallback for event handlers

All event callbacks in App.tsx are wrapped in useCallback. Stable function references between renders prevent unnecessary re-renders in child components that receive them as props.

React.memo on TransactionList and TransactionForm

Both components are wrapped in React.memo. They skip re-render when their props are unchanged — especially beneficial when parent state changes for unrelated reasons (e.g. modal opening).

Code splitting with manualChunks

vite.config.ts defines manualChunks to split vendor libraries into separate chunks: vendor-react, vendor-query, vendor-utils, vendor-forms. App code changes frequently; vendors rarely do. The browser can reuse cached vendor chunks across deploys.

Key Points for a Senior Discussion

Notable design decisions and trade-offs
Dual column for customer name: Storing customerName (original) and customerNameNormalized (canonical) elegantly solves case/space-insensitive quota counting without losing the original value for display.
long in DTO, int in entity: Receiving amountInPesos as long in the record allows validation with @Max(Integer.MAX_VALUE) without silent overflow. The service casts to int only after validation. Prevents an entire class of subtle bugs.
In-memory rate limiter (Caffeine) is intentional: Not distributed. Works for a single node. A multi-instance production setup would need Redis. The design makes it replaceable (ClientRateLimiter + RateLimitFilter are independent).
Fixed Window vs Token Bucket: Fixed Window was chosen for simplicity and straightforward implementation. Trade-off: it can allow bursts at window boundaries. Token Bucket would be smoother but more complex.
Flyway + ddl-auto: validate: Recommended pattern for production. Flyway is the schema source of truth. Hibernate validates but never modifies. Avoids surprises with automatic drops/creates.
Optimistic cache on frontend: Mutations update the React Query cache directly. This avoids a server round-trip after each operation, giving immediate visual feedback.
Package by feature: All transaction domain logic in one package. Facilitates cohesion, navigation and a potential split into a microservice if the project grows.
Java Records for DTOs: Immutable by default, no boilerplate (no getters/setters/equals/hashCode). Clearly express the intent of "data transfer only".
